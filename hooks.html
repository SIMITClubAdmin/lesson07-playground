What are Hooks?

Hooks are React's memory system for the client side. They're special functions that let you "hook into" React features (like state and lifecycle methods) from functional components.

üìú The Golden Rules

‚úÖ
Call at the Top Level

Always use hooks at the top of your component, before any returns

‚ùå
Not Inside Loops/Conditions

Never put hooks inside if statements, loops, or nested functions

‚úÖ
Only in React Functions

Only call hooks inside React components or custom hooks

‚ùå
Not in Regular JS Files

Hooks won't work in plain JavaScript files or classes

üß† Part 1 ‚Ä¢ State

useState ‚Äì Remembering Things

State & Interactivity

‚ö†Ô∏è The Problem

Normal JavaScript variables don't tell React that something changed. useState does.

‚ùå Normal Variable

What happens with let

function Counter() {
  let count = 0;

  function inc() {
    count++;
    console.log(count);
  }

  return (
    <>
      <p>{count}</p>
      <button onClick={inc}>+</button>
    </>
  );
}
Click the button:
‚úÖ count becomes 1
‚úÖ Console logs 1
‚ùå UI stays 0
Why?
  • React renders once
  • Changing a normal variable does not re-render
  • React doesn't know anything changed
‚úÖ useState

What useState actually does

function Counter() {
  const [count, setCount] = useState(0);

  return (
    <>
      <p>{count}</p>
      <button onClick={() => setCount(count + 1)}>
        +
      </button>
    </>
  );
}
Click the button:
1 setCount(1) is called
2 React is notified: "state changed!"
3 React re-runs the component
4 Virtual DOM updates
5 Real DOM patches only the text
‚ú® UI updates correctly
üí°

The BIG Idea

React only re-renders when state or props change. Normal variables are invisible to React.

‚ö° TL;DR

üìù Normal variables = forget after every render
üß† useState = remembers value, triggers UI update
‚ö†Ô∏è Must always use set..., never mutate state directly

üìê The Syntax

Destructuring the useState hook

const [count, setCount] = useState(0)
count The current value (read it)
setCount Function to update it (write it)
0 Initial value (starting point)

üìå Analogy: Sticky Note

Think of useState like a sticky note inside the component that stays stuck even if the component re-renders. The value persists!

useEffect ‚Äì Doing Things After

Side Effects

üéØ Purpose

üì° Fetching data

üìê The Syntax

useEffect(() => {
  // Code to run (the "effect")
  console.log("Component loaded!")
}, [dependencies])  // When to run

üì¶ The Dependency Array

[]

Run once (on Mount)

useEffect(() => {
  fetchData()
}, [])  // Empty!
Use case: Initial data fetch, setup
[count]

Run when count changes

useEffect(() => {
  saveToStorage(count)
}, [count])
Use case: React to changes
‚ö†Ô∏è No array

Run EVERY render!

useEffect(() => {
  // DANGER! Runs infinitely
})  // Missing []!
‚ö†Ô∏è Warning: Potential infinite loop!

useState vs useEffect

üé®

useState

The Painter

Job: Updates the screen (re-renders UI)
How: Call setCount(5) ‚Üí React re-paints
Result: User sees the number change
ü•≥

useEffect

The After-Party

Job: Runs extra code after paint
How: Waits for UI, then executes effect
Result: Side effects happen (save, fetch, log)

2
üñºÔ∏è

3
ü•≥
useEffect runs

Side effects execute after paint
